tiles-performance-comparison
東京圏における建築物3DTilesモデルの最適化前後比較
背景

プラトーからダウンロードした3D Tilesを読み込む際、ブラウズがスムーズでなく、カクつきが発生する問題がありました。

分析

3D Tilesの一部のタイルのモデルサイズが大きく、最大で50.6MBに達していました。このサイズのモデルを切り替えながら読み込む過程で、スレッドが一時的に停止する現象が発生していました。

技術的アプローチの検討

１. プラトーの3D Tilesを直接最適化
　→ 最大のタイルはルートノードであることが多く、直接分割できないため断念。

２. プラトーのCityGMLをGLTFにエクスポートして3D Tilesを再生成
　→ 公式エクスポートツールでは全てのメッシュが統合され、建物単位での分割ができないため断念。

３. FBX / OBJ モデルをGLTFに変換してからタイル分割して3D Tilesを生成
　→ 実験で可能であることを確認し、この方法を採用。

最適化ポイント

１. タイルサイズの制御
　→ 各タイルのモデル面数を5,000以下に制限。

２. KTX2テクスチャ圧縮
　→ VRAM使用量を約80%削減、メモリ使用量を約30%削減。

３. LOD（Level of Detail）の導入
　→ プラトー提供の粗モデルと精密モデルを活用。遠距離ではLOD1のみ、近づくとLOD2を読み込む方式を採用。

今後の改善予定

１. LODの違和感の解消
　→ 現状、LOD1とLOD2の差が大きく違和感あり。中間のLODを生成し、LOD1にLOD2の簡易テクスチャを貼ることで自然な遷移を実現予定。

２. LODとタイルの整合性
　→ 現状、一部エリアはLOD1のみ、他はLOD1とLOD2の混在が発生。近接時にLOD1のみの建物が消える問題があるため、LOD1のみとLOD1+LOD2のエリアを分離してタイル分割予定。

３. 過小タイルの統合
　→ 分割後に小さすぎるタイルが存在する場合、隣接タイルと統合してリクエスト数を減らし、読み込み速度を改善予定。
